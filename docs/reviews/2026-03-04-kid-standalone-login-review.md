# 子ども単独ログイン機能レビュー（2026-03-04）

対象: 直近の「子ども単独ログイン」実装

## 総評
現状の実装は **「子ども向けUIを分離」** まではできていますが、
認証・認可の実体はまだ **保護者の Supabase ログインセッション依存** です。
そのため、要件が「子どもが保護者ログイン無しで使えること」の場合は未達です。

## 主な指摘事項

### 1) 単独ログイン要件未達（保護者ログイン依存）
- 子どもPIN検証アクションが `supabase.auth.getUser()` を必須としており、未ログイン時は `/login` にリダイレクトされます。
- 子どもページ側も `is_child_in_my_family` を都度呼び、同じく保護者ユーザー文脈前提になっています。
- `/kids/login` 画面の子ども一覧取得も `getChildrenForCurrentUser()` 依存です。

**影響**
- 保護者セッションが切れると、子どもは実質利用できません。
- 「子ども単独ログイン」の期待と挙動が一致しません。

### 2) 子どもセッションCookieが生IDのみ（署名/有効期限なし）
- 子どもセッションCookieには `child_id` がそのまま保存され、署名・改ざん検知・期限がありません。

**影響**
- 現状は保護者ログイン必須チェックで一部防御されていますが、設計として脆弱です。
- 将来「本当の単独ログイン」に寄せると、そのままではなりすましリスクが顕在化します。

### 3) 子ども導線から遷移する詳細ページが保護者向け実装依存
- 子ども本だな/ホームから `/records/[recordId]` に遷移していますが、当該ページはクライアント側で Supabase ユーザー前提の取得を行う構成です。

**影響**
- 保護者未ログイン状態では、子ども導線で「遷移はできるが表示/操作できない」状態が起きうります。

## 優先度つき改善提案

### P0（最優先）
1. 子ども認証を保護者セッションから分離する。
   - 例: `child_auth_methods` をサーバー側で直接照合し、子ども専用セッションを確立。
2. 子どもセッションCookieを署名付き（HMAC/JWT等）＋期限付きに変更。
3. 子どもページ/アクションの認可を「子どもセッション起点」に統一。

### P1
4. 子ども導線の詳細ページを「子ども閲覧専用ページ」に分岐、または既存ページを子どもセッション対応。
5. PINロック時の残り時間表示など、再試行可能時刻をUIに出してUX改善。
