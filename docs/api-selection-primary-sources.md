# 書籍検索API選定メモ（一次情報ベース / 2026-03）

本メモは、読書記録アプリの API 選定について、**公式ドキュメント・公式規約（一次情報）を最優先**で判断するための社内メモです。  
現時点の実装（Google Books / NDL / OpenBD 併用）を前提に、将来の商用化へ向けた意思決定を安全に進めることを目的とします。

---

## 1. 結論（現時点の実務方針）

- **短期（今すぐ）**: 現行の多段検索を維持。
  - ISBN: OpenBD → Google Books
  - タイトル: Google Books → NDL（必要に応じて統合）
- **中期（商用化準備）**: API ごとの利用規約監査を定期運用し、データソース依存を可視化する。
- **長期（収益化拡大）**: 単一API依存を避け、フォールバック可能な構成を維持する。

> 補足: 本メモは「法的助言」ではありません。課金モデル導入前に、必ず最新版規約と必要に応じて専門家レビューを行うこと。

## 3. アーキテクチャ判断基準（商用化を見据えた優先順）

1. **規約適合性**: ユーザー課金・広告・アフィリエイトに対して禁止/制限がないか
2. **継続性**: API 廃止や要件変更時に代替可能か（ロックイン回避）
3. **運用性**: レート制限下でも UX を維持できるか（キャッシュ/キュー実装余地）
4. **品質**: 日本語書籍のカバレッジ、書影、ISBN 精度
5. **速度/コスト**: レスポンスと運用コスト（同期ジョブ、監視、失敗時対応）

---

## 4. 実装・運用で必須にするガードレール

- **データソース識別の保存**: 取得元（google / ndl / openbd / rakuten など）をレスポンス/DBで追跡可能にする
- **規約監査ログ**: 「いつ・どのURLの規約版を確認したか」を残す
- **表示要件の分離実装**: APIごとのクレジット表示をUIで切り替える
- **削除要求オペレーション**: ISBN 指定で非表示化できる運用フローを用意
- **フォールバック監視**: API障害・空振り率・429率を計測し、依存比率を可視化

---

## 5. 現行実装との整合（このリポジトリ）

現状の実装は、すでに多段検索を採用している。

- ISBN 検索: OpenBD 優先、失敗時 Google Books
- タイトル検索: Google Books と NDL を取得・統合
- 最終フォールバック: 自前カタログ検索

このため、当面は「全面置換」より、**規約監査と運用計測を先に固める**方が安全。
