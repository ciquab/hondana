# 年齢切替のための「生まれ月入力」導入提案

## 結論
- **中長期的には「生まれ月（できれば生年月）」入力を導入する方が望ましい**です。
- ただし、既存データ移行と保護者の入力負担を考慮し、段階導入を推奨します。

## 背景
現行は `birth_year` のみで年齢モードを自動判定しており、月日が不明です。
そのため境界年齢（8〜10歳前後）で、実年齢と表示モードがズレるリスクがあります。

## 期待できる改善
1. **判定精度向上**
   - 誕生日を跨いだタイミングで正しく junior/standard を切り替え可能。
2. **実機との不一致減少**
   - 「○年生まれなのに漢字表示」という問い合わせを減らせる。
3. **運用の単純化**
   - 現在の暫定補正（年齢閾値のバッファ）に依存しなくてよくなる。

## 懸念点
1. **入力負担**
   - 家族設定で追加入力が必要。
2. **データ移行**
   - 既存 child の `birth_year` しかない行に対する扱いが必要。
3. **プライバシー配慮**
   - 日付粒度をどこまで持つか（生月のみ／生年月日）を設計で明確化する必要。

## 推奨方針（段階導入）
### Phase 1（短期）
- 既存の `birth_year` 仕様は維持。
- UI上で「年齢判定は生年ベース」の注記を追加。
- `age_mode_override` の手動切替導線を明示。

### Phase 2（本対応）
- `birth_month`（必要なら `birth_day`）カラムを追加。
- 判定ロジックを「現在日付 - 生年月日」に更新。
- 既存データは `birth_month=null` で移行し、未設定時は従来ロジックにフォールバック。

### Phase 3（移行完了）
- 保護者に任意で生月入力を促す（設定画面で非ブロッキングに更新依頼）。
- カバレッジが十分になったら年ベースフォールバックを縮小。

## 受け入れ基準
- 生月入力済み child は誕生月に mode が期待どおり切り替わる。
- 生月未入力 child は従来互換で表示崩れがない。
- 境界ケース（8/9/10歳）を自動テストで担保できる。
